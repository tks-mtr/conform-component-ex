> 本プロジェクト（Redis PubSub リアルタイム通知）における、API設計書の標準構成を定義する。
> 各API設計書はこのテンプレートに従って記載する。

---

## 構成一覧

| # | セクション | 内容 |
|---|-----------|------|
| 1 | 概要 | APIのビジネス目的と処理の全体像 |
| 2 | パス | エンドポイントURL |
| 3 | メソッド | HTTPメソッド |

## 各セクションの記載要領

### 1. 概要

APIの物理的なエンドポイントとビジネス目的を明示する。
どのレイヤーが担当するか（BFF / NestJS）も記載する。

```
- 担当レイヤー: BFF（React Router v7）/ NestJS
- 目的: このAPIが解決するユースケースを1〜2文で記述
- 呼出元: どこから・どのタイミングで呼ばれるか
```

### 2. パス

```
/path/to/endpoint
```

パスパラメータがある場合は変数名と型を記載する。

```
/notifications/:jobId/read
  └─ jobId: string — 対象ジョブの識別子
```

### 3. メソッド

使用するHTTPメソッドを記載する。

```
GET / POST / PUT / PATCH / DELETE
```

### 4. アクター制御

アクセス可能なロールと認証方式を記載する。

```
- 認証方式: セッションCookie / サービス間トークン / 認証不要
- アクセス可能なロール: （例: ログイン済みユーザー / 内部サービスのみ）
```

### 5. リクエスト

#### 5-1. ヘッダー

| ヘッダー名 | 必須 | 値・説明 |
|-----------|------|---------|
| Content-Type | 条件付き | `application/json`（ボディあり時） |
| Cookie | 条件付き | セッションCookie（認証が必要なAPI） |

#### 5-2. パラメータ

**パスパラメータ**

| パラメータ名 | 型 | 必須 | 説明 |
|------------|---|------|------|
| - | - | - | - |

**クエリパラメータ**

| パラメータ名 | 型 | 必須 | 説明 |
|------------|---|------|------|
| - | - | - | - |

#### 5-3. ボディ

データモデルとJSONサンプルを記載する。

```ts
// データモデル（TypeScript型定義）
type RequestBody = {
  fieldName: string  // 説明
}
```

```json
// サンプル
{
  "fieldName": "value"
}
```

### 6. レスポンス

#### 6-1. ヘッダー

| ヘッダー名 | 値 |
|-----------|---|
| Content-Type | `application/json` / `text/event-stream` 等 |

#### 6-2. 正常系

| ステータスコード | 説明 |
|---------------|------|
| 200 OK | 取得成功・更新成功 |
| 201 Created | リソース作成成功 |
| 204 No Content | 処理成功・レスポンスボディなし |

#### 6-3. ボディ（正常系）

```ts
// レスポンスモデル
type ResponseBody = {
  fieldName: string
}
```

```json
// サンプル
{
  "fieldName": "value"
}
```

#### 6-4. エラー（異常系）

**共通エラーレスポンスモデル**

すべてのAPIはエラー発生時に以下の統一構造を返す。

```ts
type ErrorResponse = {
  status: number
  messages: {
    key: string     // エラーが発生したパラメータの物理名（例: "jobId"）
    message: string // ユーザー・ログ向けの具体的なエラー内容
  }[]
}
```

```json
// サンプル
{
  "status": 404,
  "messages": [
    { "key": "jobId", "message": "指定されたjobIdが存在しません" }
  ]
}
```

**ステータスコード定義**

| コード | 説明 | 発生条件 |
|--------|------|---------|
| 400 | Bad Request | リクエスト形式不正・必須パラメータ欠落・バリデーションエラー |
| 401 | Unauthorized | セッション無効・タイムアウト |
| 403 | Forbidden | アクセス権限不足 |
| 404 | Not Found | 指定IDのリソースが存在しない・削除済み |
| 409 | Conflict | ユニーク制約エラー |
| 422 | Unprocessable Entity | ボディ構造の不整合 |
| 500 | Internal Server Error | サーバー内部エラー（DB・Redis障害等） |

### 7. 関連テーブル

処理に関与するDBテーブルを記載する。

| テーブル名 | 操作 | 備考 |
|-----------|------|------|
| job_results | SELECT / INSERT / UPDATE | - |

### 8. 処理概要

#### DB関連チェック

処理前に実施するデータ検証ロジックを記載する。

```
- 存在チェック: 指定jobIdのレコードが job_results に存在するか
- ステータスチェック: 現在のステータスが操作可能な状態か
```

#### 処理フロー

```
1. リクエストのバリデーション
2. DB関連チェック
3. 【トランザクション開始】
   - DB操作（INSERT / UPDATE 等）
4. 【トランザクション終了】
5. Redis PUBLISH（必要な場合）
6. レスポンス返却
```

> **トランザクション対象:** データの整合性を保つ必要があるDB操作の範囲を明示する。
> 現プロジェクトはSQLite（WALモード）を使用。書込みロックの競合に注意。
