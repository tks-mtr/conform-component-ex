# Functional Requirements

## 機能概要

1. **非同期処理の検知と永続化** 外部システムから結果ファイルが連携された後、ファイル監視システムが更新を検知する。検知後、対象のユーザーID（`user_id`）と共にDBのステータスを更新し、通知用の未読フラグ`is_notified = false`をセットする。
2. **メッセージブローカーへの配信** DB更新後、ファイル監視システムからRedisの特定のチャンネルへ、ユーザーIDを含めた予約完了のペイロードをPUBLISHする。
3. **BFFからのプッシュ通知** Redisを SUBSCRIBE しているBFF（Remix/NestJS）がメッセージを受信し、ペイロードのユーザーIDを基に該当するクライアント（ブラウザ）へSSE（Server-Sent Events）を用いて結果をプッシュ通知する。

## 画面機能概要

ログイン後の全画面を対象に、ユーザーの操作を妨げることなく、バックグラウンドで予約完了のリアルタイム監視を行う。
通信切断や画面リロード時にも通知の取りこぼしが発生しない堅牢なUXにする。
1. **グローバル監視とリアルタイム表示** ログイン後の共通レイアウト（React Router v7）にて常にSSEコネクションを維持し、BFFからのイベントを受信した瞬間に、画面上に結果通知モーダルをオーバーレイ表示する。
2. **リロード時のすれ違い（欠損）防止** 画面の初回描画時およびリロード時（loader 実行時）に、ログイン中のユーザーに紐づき、かつ「未読通知フラグ`is_notified = false`」または「処理中`status = 'processing'`」を持つDB上のデータを取得する。SSE切断中の数秒間に完了した処理結果も確実にサルベージして画面に表示し、処理中のデータがある場合はSSEの接続を再開する。

## 既読化（通知フラグ更新）APIの定義

画面リロード時に未読フラグ`is_notified = false`を持つデータを取得・表示する仕様において、同じ完了モーダルが何度も表示されることを防ぐためのエンドポイント。
*    **エンドポイント例:**  `POST /api/`
*    **呼び出しトリガー:**  クライアント（ブラウザ）上で、ユーザーが結果通知モーダルを閉じる（確認する）操作を行ったタイミング。
*    **バックエンド処理**  パスパラメータに合致するDBレコードの`is_notified`フラグを`true`に更新する。
*.   **フロントエンド制御**. もしこの既読化APIの通信がエラーになった場合は、次回リロード時に再度モーダルが表示されてしまうため、バックグラウンドでのリトライ処理を組み込むことが望ましい。

## エラー発生時の再接続・復旧ポリシー

1. **クライアント・BFF間のSSE切断と再接続**
    * **自動再接続の利用**. ブラウザ標準の`EventSurce`APIが持つ自動再接続機能（デフォルトのバックオフ処理）を利用し、ネットワーク瞬断時には自動でBFFに対する再接続を試みる。
    * **切断中のデータ欠損（すれ違い）防止**. 通信が瞬断している数秒〜数十秒間に外部システムの処理が完了し、BFF（Redis）へPUBLISHされてしまうケース（Pushイベントの空振り）を想定する。
    * これを防ぐため、**BFFのSSEエンドポイント（`/api/sse/notifications`）は接続を確立しストリームを返し始める最初のタイミングで1回だけ、未読一覧API（DB）から過去の未完了分のデータを取得**し、初回データとしてクライアントへ流し込む仕様とする。
    * これにより、ユーザーが手動で画面リロード（loader実行）を行わずとも、EventSourceの自動再接続が機能した瞬間に画面のモーダルが復帰して通知を受け取ることができる。
2.  **画面リロード・再アクセス時のセッション復旧**
    * ユーザーが処理中に誤って画面をリロードした場合や、別画面から再アクセスした場合は、初回描画時（loader）にDBを照会する。
    * 基本的にはログイン中はSSEの常時接続を行うが、リロード時などの再接続のタイミングでは、「処理中」または「未読」のデータが存在する場合をトリガーとして確実にSSEの再接続（または継続）を行う。これにより、不要なリソース消費を最低限に抑えつつ欠損を防ぐ。
3.  **既読化API通信エラー時のフォールバック**
    * ユーザーが完了モーダルを閉じた際に発行する「既読化API`is_notified = false`への更新：がネットワークエラー等で失敗するケースを想定する。
    * エラーが発生した場合は、クライアント（フロントエンド）側で**バックグラウンドでのリトライ処理**を一定回数実行する。リトライが最終的に失敗した場合でも、次回リロード時に再度モーダルが表示される仕様（フェイルセーフ）として扱うことで、通知の未確認状態を安全に回避する。

## インフラ・基盤要件

本BFF（React Router v7 / Node.js）は、**AWS ECS (Fargate)** など常駐型のサーバー/コンテナ環境で稼働することを前提とする。
VercelのようなServerless Functions環境に見られる「数十秒でのコネクション強制切断」の制約を受けないため、SSEを用いた持続的なコネクション維持を想定したアーキテクチャとする。
